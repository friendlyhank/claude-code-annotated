## 实现原理与验证方式

**实现原理**：
- FileWriteTool 通过 buildTool 创建，核心是 validateInput + call 两个异步方法
- validateInput 执行前置检查链：密钥检测（TODO）→ deny 规则 → UNC 路径安全 → 文件存在检查 → 先读后写检查 → 修改时间检查
- call 执行原子读写：mkdir → fileHistory备份（TODO）→ readFileSyncWithMetadata → 修改时间比对 → writeTextContent('LF') → LSP通知（TODO）→ 更新 readFileState → 生成 patch → 返回结果
- 关键设计：writeTextContent 强制使用 'LF' 行尾，因为模型在 content 中发送了明确的行尾，不应被旧文件的行尾风格覆盖（之前的实现会在 Linux 上覆盖 CRLF 文件时产生 \r 损坏）

**验证方式**：
- 输入 `file_path: "/tmp/new.txt", content: "hello"`，文件创建成功，输出 `type: 'create'`
- 输入已存在文件路径，先 ReadTool 读取后写入，输出 `type: 'update'`，包含 structuredPatch
- 输入未先读取的已存在文件路径，validateInput 返回 errorCode 2
- 输入读取后被修改的文件路径，validateInput 返回 errorCode 3 或 call 抛出 FILE_UNEXPECTEDLY_MODIFIED_ERROR

## 遇到的问题与风险

1. FsImplementation 接口需要扩展 statSync/lstatSync/realpathSync/mkdir 等方法才能支持 readFileSyncWithMetadata 和 safeResolvePath
2. PermissionDecision 类型签名兼容性：checkWritePermissionForTool 的 tool 参数需要兼容 BuiltTool 的 getPath 签名

## 今日学习收益

1. FileWriteTool 与 FileEditTool 的核心差异：Write 是全量替换，Edit 是精确字符串替换；Write 强制 LF 行尾避免跨平台损坏
2. 临界区原子性保证：readFileSyncWithMetadata 是同步的，在"读取-比对-写入"之间无 yield，防止并发编辑交错
3. Windows 时间戳兜底：mtime 可能因云同步/杀毒变化但内容未变，对完整读取做内容比较避免误判
4. diff.ts 的 & 和 $ 转义：这两个字符会让 diff 库解析错误，需要替换为 token 后还原
